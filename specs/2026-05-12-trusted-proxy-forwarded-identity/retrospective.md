# Phase 28 Retrospective — Trusted-Proxy Forwarded Identity Authentication

Written after PR #390 merged. The spec landed the happy-path cleanly; what surfaced during implementation and review (Copilot's four inline findings plus self-review) were edge cases at the seams with existing code. Captured here so the broader patterns reach `specs/lessons.md` and future auth-chain phases.

## Deviations from the spec

- **`plan.md` Group 3 step 5 — auth-chain ordering let the trusted-proxy header silently upgrade `tk_` agent keys to admin.** Plan ordered "JWT cookie → trusted-proxy header → agent key", so a request carrying both `X-Jentic-API-Key: tk_…` and the proxy header was authenticated as an admin human session before the agent-key path ran. Fix `b296422` added a `tk_`-prefix short-circuit at the top of `_trusted_proxy_response`. The "two-actor authentication invariant" constraint in `requirements.md` named the boundary; the plan's ordering violated it.
- **`plan.md` step 5a — `/user/me` JWT-cookie fallback still returned an arbitrary JIT row.** Plan addressed the `LIMIT 1` problem only for the new proxy-auth path (via `request.state.username`); the legacy JWT-cookie fallback (`SELECT username FROM users LIMIT 1`) was left intact, so JWT admin sessions could resolve any JIT user's name. Fix `3701cdf` narrowed the fallback to `WHERE password_hash IS NOT NULL ORDER BY created_at ASC LIMIT 1`; follow-up `6eb088a` then embedded the username in JWT claims and set `request.state.username` for JWT sessions. That second commit also surfaced a deploy concern the spec never raised — JWTs in users' cookies survive a deploy, so the new `make_jwt(secret, username)` signature had to keep accepting tokens issued under the old shape. `tests/test_jwt_username.py::test_legacy_jwt_me_falls_back_to_db` pins the backward-compat path; without it, every logged-in admin would have been logged out at deploy time.
- **`plan.md` Group 1 step 1 — migration omitted backfill on upgrade and row cleanup on downgrade.** Plan specified `add column` + `drop NOT NULL` but didn't say to backfill `created_via='local'` for existing non-NULL-password rows (left `created_via=NULL`, breaking `docs/auth.md`'s "reliably distinguish local vs trusted-proxy" claim) or to delete `password_hash IS NULL` rows before restoring `NOT NULL` on downgrade (table-recreate would fail once JIT rows existed). Fix `d607cd2` added both.
- **Spec missed the SPA account-creation flow despite asserting "no UI changes".** `requirements.md` Out of Scope said "SPA UI changes — no 'logged in via proxy' badge, no new pages, no `/user/me` response-shape additions … the SPA's existing 200-with-`logged_in: true` path is sufficient." That was true for the *steady-state* SPA, but missed the *bootstrap* path: `ui/src/hooks/useAuth.ts` reads `/health` and exposes `isSetupOrAccountRequired = (status === 'setup_required')`; `ui/src/pages/SetupPage.tsx` renders the first-run account-creation wizard from that flag. On a fresh instance with both proxy env vars set, JIT provisioning had already created the user, but `/health` counted local-account rows and still returned `status: setup_required` — so proxy-authenticated users were redirected to the creation wizard, which would have forced them to register a redundant local password account. Fix `cfe9502` skips the `setup_required` branch when both proxy env vars are set; no UI code changed because the backend signal change was sufficient. The flow lives outside `auth.py`, so `plan.md` / `validation.md` never reached it, and the "no UI changes" framing in Out of Scope obscured that the UI's *control surface* (not just its code) was load-bearing.
- **`requirements.md` Decisions and `plan.md` Group 4 disagreed on the `X-Forwarded-Prefix` env gate.** Decisions said "both env vars required to activate"; Plan step 7 said "gate when `JENTIC_TRUSTED_PROXY_NETS` is set". The implementation followed the activation contract (both required), so the README description (which followed Plan) was wrong. Fix `87b5420` / `0fbf321` realigned docs to code; the underlying inconsistency was in the spec itself.
- **`plan.md` step 7 pseudocode missed the ASGI `scope["client"] is None` case.** Plan wrote `scope.get("client", ("", 0))[0]`. The ASGI spec allows `scope["client"]` to be *present but `None`* (e.g., Unix-socket transport), so the `.get` default never fires and `None[0]` raises `TypeError`. Fix `ba8819a` defended against it; small fix, but the lesson is that plan pseudocode shouldn't be pasted as-is.
- **The "every JIT identity is admin" privilege model was implicit, not documented.** `plan.md` step 4 set `request.state.is_admin = True` for JIT sessions "mirroring `_human_session_response`". Mechanically correct, but operationally a fan-out from "one root operator" to "every identity in the SSO directory is an admin of the Jentic instance" — a trade-off operators need to see before deploying. `docs/auth.md` originally said only "session granted"; clarification `ab2f459` added the explicit sentence *"The provisioned session carries full admin rights — every identity the reverse proxy forwards is trusted as an administrator."* The same commit also pinned the "no DB write from untrusted-peer + header" property explicitly (previously inferable from "log WARN, return None" but not stated). Both clarifications belonged in `requirements.md` and `docs/auth.md` from day one; relying on readers to infer them from plan-step code wasn't enough for an auth feature.

## Root cause

The spec was thorough on the new path in isolation (trusted peer + header → JIT auth) and on the security boundary (peer IP, never `X-Forwarded-For`). It was thinner at the **integration boundary with existing code**: the implications of multiple `users` rows on every legacy single-user query, the auth-chain ordering against every *other* path (not just the insertion point), the SPA's account-creation wizard flow driven by `/health`, the downgrade contract once the new column semantics are live, and the in-flight JWT compatibility window across the deploy. The spec was also thin on **operator-facing implications**: it specified `is_admin = True` in plan pseudocode but never stated the resulting fan-out ("every SSO identity becomes an admin") in prose — a property operators must weigh before deploying. The "no UI changes" framing in Out of Scope was technically right (no `ui/src/**` files needed edits) but obscured the fact that the SPA's *first-run flow* was load-bearing — the backend signal it consumes had to change, and only post-merge review caught it. Two additional sources of friction: one self-inconsistency between Decisions and Plan on env-gate conditions, and one place where plan pseudocode was specific enough to copy but didn't model an edge case in the underlying ASGI spec.

## Lesson for future specs

- When a spec adds a new path to the auth chain, `plan.md` must explicitly enumerate **precedence against every existing path, including the multi-credential matrix** (e.g., "what happens when the request carries both `tk_` and the new header?"), not just the insertion point. Express the two-actor invariant as an ordering proof, not a label.
- When a schema change introduces a new row type into a previously single-row table, `plan.md` must include a "grep `FROM <table>` and triage every call site" step. Patching only the path that *introduces* the new rows is not enough — legacy reads carry the old "one row exists" assumption.
- Adding a column to an existing table requires explicit migration sub-steps: **(a) backfill for existing rows when the column has documented semantic values; (b) cleanup of rows that would violate any constraint being restored by downgrade.** Without these, the column is unreliable in production and the downgrade is unsafe once new-shape rows exist.
- "No UI changes" is a claim about UI *code*, not UI *flow*. When a phase changes "how users get into the system", the spec must audit every SPA surface whose **control signal** is a backend endpoint touched (or made meaningless) by the phase — `/health` (drives the account-creation wizard), `/user/me` (drives `AuthGuard`), the login form's enabled state. Name those flows in `requirements.md` and decide explicitly whether they need a code change, a backend-signal change, or are genuinely unaffected.
- Before scaffolding finishes, cross-check `requirements.md` Decisions against `plan.md` env-gate conditions — they share the same vocabulary and must agree, otherwise docs and code drift along whichever side the implementer follows.
- Plan pseudocode is documentation, not paste-ready code. Describe behaviour ("read the ASGI peer IP defensively, treating absent or `None` as unknown"); leave the safe-access idiom to the implementer.
- When a phase changes *who counts as a human admin*, `requirements.md` and the user-facing `docs/auth.md` must state the privilege scope in prose, not just `is_admin = True` in plan pseudocode. "Mirrors today's behaviour" hides the fan-out when today is a single-root-operator model and tomorrow is "everyone in the SSO directory". Operators reading the deployment guide must see the trade-off before they configure the env vars.
- When a phase changes session-token shape (JWT claims added/removed/renamed, cookie format, bearer-token encoding), the spec must include an **in-flight token compatibility** clause: tokens already in users' cookies survive the deploy, and the receiving code must either tolerate the old shape (graceful fallback) or force re-login (cookie invalidation on deploy). `6eb088a` got this right organically by keeping the DB fallback when the JWT lacks a `username` claim, but the spec didn't model the migration window — future phases should call it out explicitly in `requirements.md` Decisions.

## Promotion candidate

**yes** — the rule "the `tk_` agent-key path must short-circuit before every human auth path that can set `is_admin=True`" is a load-bearing invariant of the two-actor authentication boundary, not a phase-specific reminder. It belongs in `specs/tech-stack.md` as a constraint that every future auth-chain change must respect.
